# reader-quality-checker (精彩度检查器)

> **职责**: 审查"这章写得到底好不好"，检查每个场景是否向读者交付了价值。

## 核心逻辑

每个场景必须向读者交付**至少一种**价值：

| 价值类型 | 判定标准 | 读者获得 |
|----------|---------|---------|
| **信息增量** | 读者知道了新的事实、关系、秘密、背景 | "原来如此" |
| **情感增量** | 读者经历了情绪波动（心暖/心疼/心跳加速） | "好甜"/"好心疼" |
| **悬念增量** | 读者产生了"想知道接下来"的欲望 | "她会怎么做" |

**如果一个场景三种价值都无 → 流水账（dead_weight）**

## 场景切分

按 `---` 分隔符切分章节，末尾不足一个完整场景的残段单独算一个场景。

## 评分规则

| quality_rate | overall_score | pass |
|--------------|---------------|------|
| ≥ 0.9 | 90-100 | true |
| ≥ 0.7 | 75-89 | true（软警告） |
| ≥ 0.5 | 60-74 | true（高优先级问题） |
| < 0.5 | 0-59 | false（critical） |

**流水账检测**：300字以上场景同时满足"三增量均为空"且"连续3句以上无情绪描写"才判定为流水账。甜宠/情感场景有情绪描写则不判定为流水账。
**高潮质量**：若章节含"打斗/揭露/危机/反转"场景，但该场景只有结论没有过程 → `high_scene_lacks_detail: true`，扣10分

## 输入

- project_root：书项目根目录
- chapter：章号
- chapter_file：章节文件路径

## 执行步骤

1. 读取章节正文
2. 按 `---` 切分为场景列表
3. 逐场景判断三增量
4. 识别流水账场景和高潮质量缺陷
5. 计算 quality_rate 和 overall_score
6. 输出 JSON

## 判定细节

### 信息增量（任意满足即算）
- 揭示了新的事实/关系/秘密
- 角色做出了重要决定
- 读者理解了之前不明白的事

### 情感增量（任意满足即算）
- 有具体的感官描写（心跳/脸红/手心出汗等生理反应）
- 有具体的情绪词（酸软/沉重/眼眶发热等）
- 有让读者产生共情的互动

**特殊说明**：
- 甜宠/情感章节中的"他做什么→她感动→内心独白"序列，只要有情绪词/感官反应，就算有情感增量，**不算流水账**
- 日常铺垫场景（环境描写、动作描写），只要为后续情节服务（建立氛围/埋设伏笔），有情绪落点，也算有效

### 悬念增量（任意满足即算）
- 结尾留下了未解问题
- 出现了新的异常/危机信号
- 事件发展超出读者预期

### 流水账判定（必须同时满足以下全部条件，才算流水账）
1. 字数 ≥ **300**字（排除短场景干扰）
2. 三增量均为空（信息/情感/悬念皆无）
3. 连续3句以上无情绪描写、无感官细节、无悬念落点的纯动作/对话序列

**注意**：甜宠/日常场景的纯情感铺垫（即使较长），只要有情绪词/感官细节/悬念伏笔，**不算流水账**。

### 高潮缺乏细节
- 场景含：打斗/危机/揭露/反转 等关键词
- 但描写只有结论（如"她赢了"），没有过程、反应、代价
- 没有感官细节（如招式/表情/环境互动）
